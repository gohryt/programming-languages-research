# Research Index

This repository collects language-design research as a corpus of structured records under `data/`, with a JSON Schema defining record shape, a CLI for editing and validating, and a static website for browsing.

The corpus is intended for any programming-language implementer. Keep records language-agnostic — prefer phrases like "a new language", "a compiler", "a runtime", or "a language server" over project-specific recommendations.

---

## Repository Layout

```
data/                       # research records, one JSON file per record
tags/                       # optional tag descriptors, one JSON file per documented tag
schema/record.schema.json   # record schema
schema/tag.schema.json      # tag-descriptor schema
scripts/research.js         # CLI: create / list / show / update / summarize
summary/index.html          # static site that renders summary/data.json
summary/data.json           # generated by `summarize`; do not edit by hand
README.md                   # this file
```

There are no per-domain `.md` files at the root anymore. Every former document (PARSERS, COMPILERS, REPRESENTATIONS, TYPES, CONCURRENCY, MEMORY, MODULES, PACKAGING, DEBUGGERS, TRACERS) was migrated into `data/` records.

## Tooling

`scripts/research.js` is the canonical CLI for working with the corpus.

| Command | Purpose |
|---|---|
| `node scripts/research.js create <kind> <id>` | Scaffold a new record. `--name`, `--aliases`, `--summary`, `--edit` are supported. |
| `node scripts/research.js list [--kind --tag a,b --tag-any a,b --match --verbose]` | List records by filter. `--tag` is intersection (every tag must match); `--tag-any` is union. |
| `node scripts/research.js list --sections [--tag a,b]` | List sections rather than records. |
| `node scripts/research.js list --group-by AXIS [--tag a,b]` | Group sections by tag values along the named axis (e.g. `--group-by family`). Reproduces a comparison-table view from the corpus. |
| `node scripts/research.js show <id>[#section]` | Print a record or one section. |
| `node scripts/research.js update <id> --add-section / --set-section-content / --add-ref / ...` | Programmatic edits. |
| `node scripts/research.js summarize --check` | Validate every record against the schema and resolve every cross-ref. |
| `node scripts/research.js summarize` | Validate, then regenerate `summary/data.json`. |

After any edit, run `summarize --check`. After accepted edits, run `summarize` so the static site reflects the new state.

## Record Shape

Records are JSON files under `data/`, validated against `schema/record.schema.json`.

Top-level fields:

- `kind` — one of `language`, `framework`, `tool`, `runtime`, `concept`, `library`.
- `name` — display name.
- `aliases` — strings the website uses for auto-linking from other records' content.
- `summary` — one-or-two-sentence orientation rendered in record listings.
- `sections` — array of `{ id, title, tags, content, refs?, sources? }`. Section `id`s are kebab-case, unique within the record.
- `sources` — top-level `{ label?, url }` array for record-wide references.
- `provenance` — optional `{ createdAt, updatedAt }` ISO timestamps.

Inside a section, `refs` is an array of `{ target, type }`. The `target` is either `record-id` or `record-id#section-id`. The `type` is a kebab-case label (`related`, `consumed-by`, `contrasts-with`, `implementation-of`, etc.) the site uses to categorize the link.

## Tags

Tags are the corpus's cross-cutting view layer. A tag is just a kebab-case string in a section's `tags[]` array; appearing there is enough to make the section show up under the tag's page on the static site. There is **no central tag list** — anyone can introduce a new tag by tagging a section with it.

### Tag form

Two equivalent forms are accepted:

- **Plain:** `compiler`, `ebpf`, `jvm`, `tradeoffs`. Use these for technology, language, or property tags that don't need an axis.
- **Axis-prefixed:** `axis:value`, e.g. `family:direct-execution-interposition`, `off-cost:zero`, `granularity:per-event`. Use these when the tag is one value along a comparison axis the static site can group by.

The schema regex is `^[a-z0-9]+(?:-[a-z0-9]+)*(:[a-z0-9]+(?:-[a-z0-9]+)*)?$`. Only one optional colon is allowed.

### Tag descriptors

Optionally, a tag can have a descriptor file under `tags/<id>.json` documenting it. The descriptor is documentation, not enforcement — a tag without a descriptor still works, it just renders a plain listing page on the website.

Descriptor schema: `schema/tag.schema.json`. Required fields are `tag` (the literal tag string) and `name`. Optional fields:

- `description` — free-form prose explaining when to apply the tag.
- `axis` — the axis prefix (e.g. `family`, `off-cost`). Used by the static site's axis-grouped views.
- `aliases` — older or near-duplicate tag strings now merged into this canonical form. The site treats them as redirects.
- `parent` — another tag, for hierarchical organization.
- `examples` — record ids that exemplify the tag (advisory; the real listing is computed from sections).

### Adding a tag

1. Apply the tag to relevant sections' `tags[]` arrays.
2. *(Optional)* drop a descriptor under `tags/<filename>.json`. Filename is the tag string with `:` replaced by `-`.
3. Run `summarize --check`. The CLI surfaces the new tag automatically — no central list to update.

### Adding an axis

An axis is just a colon prefix that several tags share. To start a new axis, use it in tag values (`my-axis:foo`, `my-axis:bar`) and optionally drop descriptor files. The static site's "Axes" navigation will pick it up after the next `summarize` run. There is no central axis registry.

### Convention: cost and granularity tiers

The 28 chapter umbrellas use these axes; the rest of the corpus is welcome to adopt them or invent its own. They are conventions, not requirements.

| Axis | Tier values | Meaning |
|---|---|---|
| `domain` | `tracing`, `debugging`, `profiling`, `observability` | Topic cluster. |
| `family` | open-ended (e.g. `direct-execution-interposition`, `cooperative-safepoints`, `record-replay`, …) | Mechanism family — the cross-cutting comparison view. |
| `off-cost` | `zero`, `low`, `medium`, `always-on` | Steady-state overhead when the mechanism is off. |
| `on-cost` | `low`, `medium`, `high`, `dominant` | Active overhead when the mechanism is on. |
| `granularity` | open-ended (e.g. `per-instruction`, `per-function`, `per-event`, `per-sample`, `per-syscall`, `per-frame`, `per-request`, …) | Unit of observation/effect. |

These map onto the comparison tables that previously lived in monolithic `.md` chapters.

### Browsing tags and axes

- CLI: `list --tag X` (intersection if comma-separated), `list --tag-any X,Y` (union), `list --group-by family` (axis-grouped sections).
- Site: `?tag=family:X` for tag pages, `?axis=family` for axis-grouped comparison views, top-nav exposes both.

## Cross-Reference Policy

Use cross-references to avoid repeating full explanations.

In record content, two forms are accepted:

- `` `data/<record-id>` `` — whole-record reference.
- `` `data/<record-id>#<section-id>` `` — specific section.

Structured links live in section `refs` arrays so the website can render them as typed links. Inline markdown links stay readable in the JSON content as well.

The good pattern in prose:

> Full representation details are in `data/rust-hir-thir-mir`; this section focuses on the compiler-pass consequences.

If a topic belongs in another record, keep only:

1. one or two sentences of local context,
2. the reason this record cares,
3. the cross-reference to the canonical record/section.

## Source Citation and Status Style

Each section's evidence URLs go in its `sources` array (`{ label?, url }`). Top-level evidence (broad references, primary papers) goes in the record's top-level `sources` array.

Use a `sources` entry when a claim is specific, surprising, implementation-dependent, numeric, historical, or likely to be checked by a reader. Broad background claims do not need an inline citation.

For date-sensitive claims, mark the date or version in the prose:

- `Status (as of 2026-04): ...`
- `Status (JDK 25): ...`
- `As of early 2026, ...`
- `Scheduled for 2026-Q2, ...`

Date-sensitive claims include standards-process status, release availability, deprecation/removal status, production-readiness, experimental/stable labels, and vendor roadmap statements. If the date is uncertain, keep the wording conservative rather than implying current truth.

Don't invent URLs. Use real searches when adding new content.

## Writing Guidelines

- Keep the research language-agnostic.
- Prefer system names and technique names over project-specific recommendations.
- Separate mechanism (how something works) from policy (when a language should choose it).
- When a topic crosses records, name the ownership boundary in the prose.
- Preserve useful trade-offs: space cost, time cost, disabled overhead, annotation burden, implementation complexity, tooling consequences.
- Mark date-sensitive claims with a date when possible, especially implementation status or standards-process status.
- Don't add broad surveys to non-canonical records — add short capsules and cross-references instead.

## Suggested Record Shape

A typical **chapter umbrella** has:

1. an `overview` section that states the scope and the design axis the umbrella explores;
2. one section per discrete mechanism, tool, or technique covered;
3. (optional) a section that maps the mechanisms to a comparative table or a design-implications summary.

A typical **per-tech record** has:

1. an `overview` section;
2. sections grouping aspects of the implementation (architecture, performance, deployment, edge cases);
3. `refs` linking back to the umbrella(s) that anchor it.

## Adding a New Record

1. `node scripts/research.js create <kind> <id>` and edit the scaffolded JSON.
2. Link from at least one umbrella record's `refs` so the new record is reachable.
3. `node scripts/research.js summarize --check` — validates schema and ref resolution.
4. `node scripts/research.js summarize` — regenerates `summary/data.json`.

## Maintenance Checklist

Before adding or expanding a record:

- [ ] Is this the canonical record for the topic, or should it cross-reference an existing umbrella?
- [ ] Does the new content duplicate another record's explanation?
- [ ] Are sources placed in the right `sources` array (section-level vs top-level)?
- [ ] Is the wording language-agnostic?
- [ ] Are date-sensitive claims marked with a date or status?
- [ ] Does any related umbrella need a new `refs` entry pointing here?
- [ ] Are sections tagged on the conventional axes (`domain:`, `family:`, `off-cost:`, `on-cost:`, `granularity:`) where they fit, plus any topic tags they merit?
- [ ] If a new tag is introduced, does it deserve a descriptor under `tags/`?
- [ ] After editing, did `summarize --check` pass and `summarize` regenerate the static site?
